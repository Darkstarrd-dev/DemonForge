# M1 文本预处理模块 — 详细设计

> 状态：草稿 v0.1（2026-06-12）。
> 本文档抽象自已验证可用的单页应用原型 `10_novel_cleaner.html`（小说章节清理工具 Pro），
> 提炼其架构、机制与交互，供 novelhelper 落地实现时使用。原型中已实测可用的参数、正则、
> prompt 原样收录；原型的已知缺陷在 §6 列出，落地时修复。

## 1. 流程总览

原型采用四步流水线，顶部步骤指示器引导：

```
① 导入文件（编码检测）→ ② 章节分割（规则 + AI 兜底）→ ③ AI 清理（并发批处理）→ ④ 审核与导出
```

novelhelper 中第④步的终点从"导出文件"改为"审核后入库"（章节写入 SQLite，状态 cleaned），
导出（合并 TXT / 分章 ZIP）保留为辅助功能。

## 2. 架构映射：单页应用 → novelhelper

| 关注点 | 原型实现 | novelhelper 落地方案 |
|---|---|---|
| 文件导入 | File System Access API，保留原始 ArrayBuffer | 前端上传至后端；后端保留原始字节，支持编码切换重解码 |
| 编码检测 | 手写 BOM 检测 + UTF-8/GBK 启发式评分 | 后端用 `jschardet` + `iconv-lite`（机制等价，更可靠）；保留"手动指定编码→重解码"交互 |
| 章节切分 | 前端正则，预设模式 + 自定义 | 后端切分服务，机制照搬（§3.2） |
| LLM 调用 | 浏览器直连 fetch（受 CORS 限制，需用户找支持跨域的 API） | 后端经 Provider 抽象层调用，**CORS 问题自然消失** |
| 并发调度 | 浏览器内 N 个 worker 递归循环 | 后端进程内任务队列，机制照搬（§3.4） |
| 多 API 节点 | localStorage 配置的节点池 | 并入 P0 的 Provider 抽象层（§3.5 是其需求来源） |
| 状态持久化 | 章节在内存、设置在 localStorage，**刷新全丢** | 全部入 SQLite：章节、状态、行级决策、重试计数 |
| 进度反馈 | DOM 直接更新 + 流式输出窗口 | SSE 推送：任务进度、活跃请求列表、当前流式输出 |

## 3. 核心机制（从原型继承）

### 3.1 编码检测与解码

- BOM 检测：UTF-8 / UTF-16LE / UTF-16BE。
- 无 BOM 时启发式评分：取前 10000 字节，按 UTF-8 多字节序列规则与 GBK 双字节范围分别计分，
  `utf8Score > gbkScore × 1.5` 判 UTF-8，反之判 GBK，不确定默认 UTF-8。
- 解码尝试链：`检测结果 → utf-8 → gbk → gb18030 → big5`，用严格模式解码，
  替换字符（U+FFFD）占比 < 1% 即接受。
- **交互**：显示检测结果；用户可手动改编码，从原始字节重新解码（不依赖已解码文本）。

### 3.2 章节切分

预设模式（原型实测正则，原样保留）：

| 模式 | 正则 |
|---|---|
| 第X章（中文/阿拉伯数字） | `^(第[0-9零一二三四五六七八九十百千万]+章.*)` |
| 第X回 | 同上，`章`→`回` |
| 第X卷 | 同上，`章`→`卷` |
| 数字章节（001章、01 章） | `^(\d{1,4}\s*章.*)` |
| 自定义 | 用户输入正则，首个捕获组为标题 |

- **预览交互**：应用切分前可预览检测结果（章节数 + 标题列表），确认后再应用。
- 选项：保留第一章之前的内容为"序章"（楔子等）。
- 无任何匹配时降级：全文作为单章。
- 标题截断至 50 字符。

**AI 拆分合并章节**（章节列表中每章可单独触发）：规则切分有时会把多章并成一章，
对该章调用 LLM 重新拆分，协议为 `>>>CHAPTER_TITLE: 标题` 行分隔，要求原文内容逐字保留。
解析失败则回滚原状态。

**样本驱动调整**（依据 `M1_raw_features.md` §2，2026-06-12）：

- 预设正则统一允许行首空白：加前缀 `^[ \t　]*`（样本存在行首带半角空格的标题行）。
- 新增预设「纯数字行」：`^[ \t　]*(\d{1,4})[ \t　]*$`（样本存在整行仅数字的标题）。
  该模式易把正文中独立成行的数字乱码误判为标题，依赖切分预览人工确认。
- 标题行内部可能穿插乱码（如 `第589三92章` 碰巧仍匹配现有字符类，但 `第58宭9章`
  之类会漏切）→ 预览中增加**章节号连续性检查**：解析各标题的章节号（阿拉伯/中文数字），
  对跳号、乱序、重复处标记可疑提示；漏切的章用既有「AI 拆分合并章节」兜底。
- 作者请假条等无标准标题行的内容（样本：`请假一天`）不参与切分，会并入前一章，
  留给清理/审核阶段处理。

### 3.3 AI 清理的批量协议

- 单请求可携带多章：以 `<<<|||CHAPTER_SEP|||>>>` 分隔，每章前缀 `===CHAPTER_ID:X===` 标记行；
  系统提示词附加严格指令（必须返回 N 段、保留标记、不得合并/遗漏）。
- **流式渐进解析**：流式输出过程中按分隔符切分，已完整的段落立即回填对应章节并标记完成
  ——用户不必等整批结束就能看到前面章节的结果。
- 返回段数不符时宽容处理：有多少用多少，单段内容 < 10 字符视为该章失败（触发重试）。
- 【落地调整】分隔符协议对模型遵循度敏感（原型为省请求次数而设计）。落地后默认**单章单请求**
  （后端无并发数顾虑），批量模式保留为可选优化。

### 3.4 动态任务队列与重试

- 双队列：`pending`（待处理）+ `retry`（重试，**优先于 pending 出队**）。
- N 个 worker 循环取活，处理完递归取下一批；暂停（轮询等待）/ 停止（标志位 + 出队回滚）。
- 每章独立 `retryCount`，上限 3 次；超限标记 `failed`，不阻塞整体。
- 章节状态机：`pending → processing → completed → accepted / rejected`（另有 `failed`、`retrying`）。
- **标记重新处理**：审核时对可疑章节打 `needsReprocess` 标记，下次启动时优先处理。
- 处理范围可选（起始章—结束章），默认全部。

### 3.5 多 API 节点池（Provider 抽象层的需求来源）

原型的多源并发模式，落地时并入 P0 Provider 抽象层：

- 节点属性：名称、URL、Key、模型、最大并发、单次章节数（批大小）、请求间隔，可单独启停。
- **节点选择策略**：过滤启用且未满载（activeCount < maxConcurrency）且间隔已到的节点，
  按 `lastRequestTime 升序 → activeCount 升序 → 配置顺序` 排序取首个（最久未用 + 最少连接）。
- 总并发 = 各节点最大并发之和。
- 429 处理：原型直接将节点移出池（全部耗尽则停止）。【落地改进】改为**冷却期后自动恢复**。
- 节点连通性测试：单测 + 批量测试（间隔 5 秒防触发限流），结果状态标记在节点上。

### 3.6 速率控制

- RPM 滑动窗口限制：维护 60 秒内请求时间戳队列，超限等待。
- 全局请求间隔（防止并发请求同时发出触发 429）：**错峰锁**——worker 取锁后发请求，
  HTTP 连接建立（onConnect）后延迟 interval 秒释放锁，实现请求起点错开而非串行化。

### 3.7 清理 Prompt

#### v2 默认版（2026-06-12 设计；机制依据见 §3.9 与 `M1_raw_features.md` §4，允许用户编辑）

```
你是一位专业的小说文本编辑。你拿到的章节来自网络流传的 TXT 文件，发布渠道在正文中
植入了大量推广 QQ 群的广告杂讯。你的任务是清除杂讯、尽可能恢复作者原文。

## 先理解杂讯的原理（这决定你能否识别没见过的变种）

这些杂讯不是写给人眼看的，而是针对"听书"（TTS 朗读）的：人眼阅读时会自动跳过
乱字符并在脑内复原句子，但朗读软件会把它们逐字念出来——广告商利用这一点，把
QQ 群号用"念出来同音"的各种写法拆碎混进正文，让听书的读者听到群号；同时夹入
不会被念出来的符号与不可见字符作干扰。由此可知杂讯的两个特征：
- 删掉它，句子恢复通顺；留着它，朗读会被打断——满足这一条的就是杂讯；
- 数字会以任意"同音/同形"写法出现：阿拉伯与全角数字（371、８９３）、汉字大小写
  （三七一、叁柒壹）、谐音字（吆=1 贰=2 伞=3 肆寺似=4 柳遛=6 柒漆气=7 扒吧疤=8
  玖久韭=9 邻零=0）、带圈/带括号数字（②、⒍、（一））、拼音（jiu、ba、liu）、
  罗马数字（IX）；"群/说/小"等关键词写作变体字（羣峮裙囷宭、説裞、仦）。

## 删除规则

1. 整块广告：加群邀请、网盘链接与密码、群号列表、盗版声明（如【本作品来自互联网…】）、
   阅读 App 推广、<img> 等 HTML 标签，整段删除。
2. 句中穿插的群号碎片与符号杂讯：连同夹带的变体字、多余空格一起删除；
   碎片常把词语或人名从中间劈开，删除后必须把词语复原——
   例：「毛小説羣89利兰」→「毛利兰」；「公  3气1漆平竞（二）9吆伊9争」→「公平竞争」。
3. 行首藏号：连续多行的行首各多出一个无关字符（纵向连读是群号或广告词）时，
   逐行删除该行首字符。
4. 平台水印：如「（看暴爽小说，就上飞卢小说网！）」这类嵌在句中的括注，整段删除；
   注意与作者本人的括号旁白区分（旁白与情节相关，水印与情节无关）。
5. 格式：把 3 行以上的连续空行压缩为 1 行，其余不改变段落结构。

## 默读测试（拿不准时的判定法）

把句子在心里默读一遍：跳过该字符串后句子才通顺、且它与情节无关，删；
它是句子的有机组成部分，保留。

## 保留红线（错删比漏删严重）

- 不改写、不润色：不改变作者的用词、标点与风格。半角省略号「...」、破折号、
  拟声词、重复标点（"痛痛痛痛！"）都是作者文风，原样保留。
- 正文里的真实数字（"一千八百万次""十六岁""明天8k"）不是杂讯，保留。
- 成片的"乱码状文字"若删掉后句子出现缺口（说明它占位着原文，是编码损坏），
  原样保留，不要尝试修复或删除。
- 作者本人的话：ps 求票、请假说明、求鲜花等，原样保留。
- 有任何疑问时保留原文。

直接输出清理后的正文，不附加任何解释。
```

> **可选注入段**（混合模式下由规则引擎阶段 1 自动产出，附在 prompt 末尾，见 §3.10）：
> 「## 本书已知广告特征（程序提取）：群号 {payloads}；品牌词 {brands}。
> 凡与上述号码呈同音对应的字符序列均为广告碎片，可放心删除。」
>

#### v1 原型版（收录存档，已被 v2 取代）

```
你是一位专业的小说文本编辑，负责清理和修正小说章节内容。

## 核心任务

### 1. 广告内容清理
需要移除的内容类型：
- 网址、链接、二维码提示
- 公众号、微信号、QQ群等推广信息
- "求收藏"、"求推荐票"、"求月票"等拉票文字

- 不改变原文的用词、风格、语气
- 不添加任何新内容
- 如有疑问，保留原文
- 直接输出清理后的文本，不添加任何解释

重要：如果遇到格式为 图片:"...",位置:"..." 的内容（其中...为任意文本），请直接整段删除。
```

> 注：v1 末行"图片占位"规则是针对特定 raw 来源的特征规则——此类来源特征现已沉淀于

### 3.8 防误删护栏

- **字符保留率检查**：清理前后去空白字符数对比，保留率低于阈值（默认 90%）显示警告
  "字符减少过多，建议检查"，提示人工细看 diff。
- LLM 输出过短（< 10 字符）判失败重试，防止模型输出解释文字替代正文。

### 3.9 规则清理引擎（不依赖 LLM；原型已实证）

> **原型**：`frontend/src/utils/ruleClean.ts`（纯 TS 零依赖，噪声库表可配置）；
> **验证**：`frontend/scripts/ruleclean-smoke.mts` 对真实 raw 样本 43 项断言全过——
> 载荷提取 5/5 无污染、删除 75 处、低置信标记 18 处，保留红线（人名、真实数字、
> 半角省略号、mojibake、作者 ps）零误删。
> 运行：`cd frontend && node --experimental-strip-types scripts/ruleclean-smoke.mts`

**核心洞察**（用户提供，2026-06-12）：广告混淆是针对 TTS（听书）的——谐音字会被
念出来、零宽符不会，目的是让听书用户听到群号。由此推出**自锚定原理**：群号必然在
文件内某处以可恢复的明文形式出现（否则广告无效），这是不依赖样本特征的通用突破口。

**通用性三层**（该路径的适用边界，回应"仅 7 本书样本能否通用"）：

| 层 | 内容 | 通用性来源 |
|---|---|---|
| 机制层 | 不可见字符剥离；结构性广告块识别；**自锚定载荷提取**；数字影子序列匹配；**自指语料复原验证**（拼接处 n-gram 在全书他处复现即确认，无需外部词典）；行首藏号窗口检测 | 广告经济学约束与文本统计，与具体书源无关 |
| 资产层 | 谐音数字映射表（按**拼音家族**泛化：读音∈ ling/yi/er/san/si/wu/liu/qi/ba/jiu 的字，分强/弱两级）、变体字表、广告词表（品牌词自提取）、水印白名单 | 按源扩充，会过时但可维护 |
| 不可达 | 全新混淆家族、mojibake 修复、无锚点的语义判断 | 留给 LLM/人工 |

**管线**（与原型实现一致）：

| 阶段 | 判定 | 误删保护 |
|---|---|---|
| 0 Unicode 层 | 剥离 U+00AD / 200B–200F / 2060–2064 / FEFF | 保留 U+3000 全角缩进 |
| 1 块级广告 | URL/网盘密码/盗版声明/加群邀请整行删；**群号行**三重判定：引号开头的对话行排除 → 明文群号+关键词，或解码 ≥6 位且去掉关键词/数字/变体/符号后剩余正文字 ≤3；提取**载荷集 P** 与品牌词；全行可映射且解码 ≥6 位（≥3 强形式）的纯杂讯行删；夹在广告行之间的短关键词行删 | 正文行天然不过"剩余正文字 ≤3"；1–4 位纯数字行视为潜在章节标题跳过 |
| 2 数字影子 | 字符→数字映射，**强**（ASCII/全角/带圈/带括号数字、大写汉字、生僻谐音、拼音、罗马）可作锚点，**弱**（小写汉字数字、常见谐音字）仅作证据 | 量级字（十百千万亿）不映射，保护"一千八百万"类真实数字 |
| 3 载荷匹配 | ①子序列：≥5 位按序命中 P + ≥2 强锚点，碎片间隔 ≤8 字符；②连续链子串：≥6 位任意形式 / ≥4 位纯强直接删；2–5 位挂起，邻接（≤2 字符）已确认噪声段才删 | 纯弱形式不得作非首位起步（防"三七""三叶"误配）；`第X章` 保护区不动；数字+单位（8k、24小时）跳过 |
| 4 变体字噪声段 | 硬变体作种子，贴边扩展（广告词跨界吸收；弱字/标点"暂存"——远端仍是噪声才并入；**引号是句子结构不可吸收**）；≥2 硬变体 / 含载荷子串 / 同行已有载荷证据 → 删；广告词连写（品牌+中转等）在有同行载荷证据时删 | 软变体（裙/説/澪有合法用途）不作种子只可被吸收；单证据段须过复原验证，不过只标记 |
| 5 行首藏号 | 连续 ≥4 行、行首字可疑率 ≥75%，且**纵读命中载荷 ≥4 位或含 ≥2 硬变体**才逐行剥离；行首杂串 ≥3 字符且含硬变体才删 | 门槛不满足只标记；窗口内正常行首字不剥 |
| 6 密集符号杂讯 | 行内孤立 ASCII 串 ≥5 处且占比 ≥12% | **只标记不删**（单一证据） |
| 7 复原验证 | 删除候选后拼接处二元组需在全文他处出现 ≥minCorpusHits 次（标点/行边界直接通过） | 整本书建议 ≥2，小样本语料用 1 |

**输出与审核**：`{cleaned, payloads, deletions[], suspects[]}`——删除结果走既有
双栏 diff + 行级决策审核（§4），suspects 在 diff 中高亮提示。

**运行模式**：

- **纯规则模式**：只应用高置信删除，suspects 留给人工 diff 审核，零 LLM 成本；
- **混合模式（推荐）**：规则层先做确定性预清理并提取载荷，LLM（§3.7 v2 + §3.10
  注入）处理 suspects 与残余——规则层削掉大部分噪声，并把已知群号喂给 LLM。

**降级策略**（载荷不可得，如广告块被上游截断）：①通用群号模式（影子中 6–11 位带
强锚点数字串）降置信处理；②跨文件噪声库——同源文件提取过的载荷持久化复用（落地
入 SQLite，原型未含）；③仅启发证据时全部只标记不删。

**实测已知局限**（须由 LLM/人工补位）：

- 样本后半（其他书源、其广告块未随样本提供）的混淆段只能标记——印证降级策略①③；
  真实整本 raw 自带广告块，自锚定可用；
- 噪声与正文的归属歧义会留残字（如「一[群893]辆装甲车」清后残留"群辆"连读）；
- mojibake、品牌字谐音变形（灵珑→笼/霖）、无锚点符号杂讯不做删除决策。

### 3.10 噪声库与 prompt 动态注入

- **噪声库**（资产层，可配置）：数字映射表（强/弱两级，按拼音家族泛化）、硬/软变体
  字表、广告词表（品牌词由引擎从『X小说群』模式自提取）、水印/声明白名单。
  落地后入 SQLite 并提供设置界面维护，跨文件共享载荷集。
- **动态注入**：规则引擎阶段 1 提取的载荷集 P 与品牌词注入 §3.7 v2 prompt 末尾的
  可选段「本书已知广告特征」，提升 LLM 对碎片的召回——这是规则层与 LLM 层的协同点。
- 原"特征追加规则"已吸收进 v2 prompt 正文。

## 4. 审核交互设计（从原型继承）

### 4.1 三视图

每章可切换：**原文 / 清理后 / 对比**。原文与清理后视图均支持整篇直接编辑保存。
章节完成后默认进入对比视图。

### 4.2 双栏对齐 diff

- 行级匹配三阶段：①去空白后精确匹配 → ②未匹配行做模糊匹配（字符集 Jaccard 相似度 > 0.5
  判为"修改"）→ ③双栏对齐输出（空行/无配对侧填占位行，保持两栏行号对齐）。
- 行分类着色：删除（红）/ 新增（绿）/ 修改（黄）/ 上下文。
- **修改行字符级高亮**：对修改行计算 LCS，原文侧将被删字符以删除线+红底标出。
- 双栏滚动比例同步；点击任一行，两侧对应行同时高亮。
- 统计栏：`删除 X 行 | 新增 X 行 | 修改 X 行` + `原文 N 字 → M 字 (保留率%)`。
- 【落地调整】手写 LCS 为 O(n²)，长段落卡顿，改用成熟 diff 库（如 `diff` npm 包）。

### 4.3 行级决策

对比视图右栏每个差异行悬浮操作钮：

- **接受**（默认）/ **拒绝**（恢复原文行，显示删除线）/ **重置**。
- **双击行内编辑**：输入框直接改该行内容，Enter 保存、Esc 取消，已编辑行打"已编辑"徽章。
- 决策存储于章节的 `lineDecisions` 映射（行号 → {action, content}）。

### 4.4 章节级操作

- 整章接受（自动跳下一章）/ 整章拒绝（丢弃清理结果）/ 标记重新处理（清空结果，下轮优先）。
- 章节列表显示每章状态图标（待处理/处理中/完成/已接受/已拒绝/待重处理）。

### 4.5 处理过程监控

- 活跃任务列表：每个进行中请求一项（含所属 API 节点标记、章节标题、状态），可点击切换查看。
- 双栏实时窗口：左侧为发送的原文，右侧为 AI 流式输出。
- 进度条 + 完成数 / 总数 + 活跃请求数。

## 5. 导出

- 合并导出：全部章节（清理后优先，否则原文）拼接为单个 TXT。
- ZIP 导出：每章一个 `序号_标题.txt`。
- novelhelper 中主出口为"入库"，导出保留为辅助。

## 6. 原型已知缺陷（落地时必须修复）

1. **行级决策未生效于导出**：原型中 lineDecisions 仅影响 diff 视图显示，导出/保存仍用
   原始 cleanedContent。落地必须实现"应用行级决策 → 生成最终文本"，入库与导出都走该结果。
2. **无持久化**：刷新页面章节与进度全丢。落地后全量入 SQLite。
3. 批量分隔符协议依赖模型遵循度：默认改单章单请求（§3.3）。
4. 字符级 diff 性能：换成熟 diff 库（§4.2）。
5. 429 永久移除节点：改冷却恢复（§3.5）。
6. 多文件导入：原型 UI 显示"N 个文件"但实际仅支持单文件，落地明确支持多文件/批量导入。
