## 2. 🚨 核心问题清单（High Priority）

### 🔴 P0-1：3D 骰子引擎未检测物理静止，也未做 slerp 校准，预设结果功能失效
`frontend/src/pages/demo-3d/Dice3DEngine.ts:95-155`

`roll()` 在创建刚体后**立即返回** `result.values`，既不等待骰子静止，也不调用 `getUpFace` 读取实际朝上面，更不对 `presetValues` 做 slerp 校准。用户开启预设并输入 `3,5` 后，UI 会显示 `[3,5]`，但 3D 视口中骰子实际朝向与结果无关。这直接违反 `docs/dice_implementation_plan.md` §3 步骤 10 的验收标准「预设模式下骰子停在指定值」。

### 🔴 P0-2：3D 骰子使用 `ball` 碰撞体，未按设计使用 `convexHull`
`frontend/src/pages/demo-3d/Dice3DEngine.ts:139`

设计文档明确要求「用几何体顶点构造凸包碰撞体 (`ColliderDesc.convexHull(vertices)`)」，并规定若 convexHull 失败需停止上报。实现改为 `ColliderDesc.ball(0.5)`，所有面数（d6/d8/d10/d12/d20）全部用球体近似，物理滚动行为不真实，且是导致「无法精确落点」后需要 slerp 校准的根因之一。

### 🔴 P0-3：`geometry.ts` 中 d12 面贴图合并逻辑错误，导致 3D d12 纹理错位
`frontend/src/game/dice/geometry.ts:310-330`

`addFaceGroups(geo, 12, 9)` 已把 d12 的 12 个五边形面各作为一个 group（共 12 个 group）。但 `applyFaceTextures` 的 d12 分支以 `gi += 3` 步进，只处理 4 个「合并组」，每组错误地吞掉 3 个五边形面。结果 d12 只有 4 种材质被应用，且材质与面的对应关系错乱。

### 🔴 P0-4：2D/3D 演示面板的 `count`/`sides`/`theme`/`physics` 参数未透传到 Scene/Engine，UI 控件无效
`frontend/src/pages/demo-2d/index.tsx:244-285`、`frontend/src/pages/demo-3d/index.tsx:276`、`Dice2DPanel.tsx:37-44`、`Dice3DPanel.tsx:36-43`

- `Dice2DPanel` 的 `count` Slider 只存在本地 state，从未写入 `game.registry`，而两个 Scene 从 registry 读 `diceCount`； Scene 内部又硬编码 `sides: 6`。
- `Dice3DPanel` 的 `count` Slider 同样只存本地 state；`handleReset` 创建引擎时硬编码 `{ count: 2, sides: 6 }`。
- 2D/3D 面板预设输入校验写死 `v <= 6`，与硬编码面数一致，但同样无法支持非 d6。

这违反设计文档 §3 步骤 7/8/9 中「UI 选项：骰子数/面数/力度/颜色/物理参数/预设结果/历史记录/动画时长/音效 全量」以及 §5「功能·预设：3D 预设模式输入 '3,5'，骰子校准后朝上面为 3 和 5」。

## 3. 🔍 逐项审计与重构建议

### 3.1　d12 面贴图映射错误

- **文件**：`frontend/src/game/dice/geometry.ts:310-330`
- **Why**：`addFaceGroups(geo, 12, 9)` 已按「一个五边形面 = 一个 group（9 个顶点）」组织。`applyFaceTextures` 的 d12 分支却假设 groups.length 是 36（每个三角形一个 group），按 `gi += 3` 合并，导致只生成 4 个合并组，材质索引与面法向量全部错位。Three.js 0.184 的 `DodecahedronGeometry` 已是非 indexed，`addFaceGroups` 之后 groups.length === 12，只需像 d8/d20 一样逐个 group 匹配即可。

**Before**
```ts
if (sides === 12) {
  // d12: 每个五边形面拆为 3 个三角形（3 个 group），合并为 1 个 group 后匹配
  const mergedGroups: { start: number; count: number; faceValue: number }[] = []
  for (let gi = 0; gi < groups.length; gi += 3) {
    const start = groups[gi].start!
    const count = groups[gi].count! + groups[gi + 1].count! + groups[gi + 2].count!
    const faceValue = matchFaceNormal(posAttr, start, faceDefMap)
    mergedGroups.push({ start, count, faceValue })
  }
  geometry.clearGroups()
  const materials: THREE.Material[] = []
  for (let i = 0; i < mergedGroups.length; i++) {
    const g = mergedGroups[i]
    geometry.addGroup(g.start, g.count, i)
    const tex = textures.get(g.faceValue)
    materials.push(new THREE.MeshStandardMaterial({ map: tex }))
  }
  return materials
}
```

**After**
```ts
// d12: addFaceGroups 已经把每个五边形面组织为一个 group（9 顶点 = 3 三角形），
// 因此与 d8/d20 一样逐个 group 匹配即可，不要额外合并。
```
（删除整个 `if (sides === 12)` 分支，让 d12 落入下方的 d8/d10/d20 公共分支。）

**验证方法**：在 `geometry.test.ts` 增加一条断言：对 d12 调用 `applyFaceTextures(...)` 后，返回的材质数组长度应为 12，且 `geometry.groups` 的 materialIndex 按 0..11 顺序排列。

### 3.2　3D 引擎 `roll()` 未等待物理结算

- **文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:95-155`
- **Why**：`roll()` 在 `for` 循环创建刚体后立即 `return result.values`。调用方 `handleDiceRoll` 拿到值立刻 `setDiceResult`。骰子仍在空中滚动，实际朝上面与返回结果可能不一致；预设值更不可能生效。必须按设计文档实现「物理滚动 → 静止检测 → 读取 `getUpFace` → 若与目标不符则 slerp 校准」的双阶段。

**Before**
```ts
const roll = async (presetValues?: number[]): Promise<number[]> => {
  const result = roller.roll({ count: params.count, sides: params.sides as never, presetValues })
  // ... 创建 mesh/body ...
  return result.values
}
```

**After（骨架）**
```ts
const roll = async (presetValues?: number[]): Promise<number[]> => {
  const targetValues = presetValues
    ? presetValues.slice()
    : roller.roll({ count: params.count, sides: params.sides }).values

  // ... 创建 mesh/body，并保存 targetValues 与 mesh 的对应关系 ...

  // 等待静止（或超时兜底）
  await waitUntilSettled(bodies, meshes, params.sides, targetValues)

  // 最终读取实际朝上面并返回
  return meshes.map((mesh) => getUpFace(mesh.quaternion, params.sides))
}
```

**关键实现细节**（供低阶模型直接实施）：
1. 在 `Dice3DEngine.ts` 内新增辅助函数 `waitUntilSettled(bodies, meshes, sides, targetValues, timeoutMs = 10000)`：
   - 每帧检查所有 body 的 `linvel().norm() < 0.1` 且 `angvel().norm() < 0.1`。
   - 连续满足 500ms（或设计文档指定的 500ms）后认为静止。
   - 静止后调用 `getUpFace(mesh.quaternion, sides)` 得到 `actualValues`。
   - 若传入了 `targetValues`，对 `actualValues[i] !== targetValues[i]` 的骰子调用 `correctDiceOrientation(mesh, getTargetQuaternion(sides, targetValues[i]), 300, () => {})` 做 slerp 校准；全部校准完成后返回 `targetValues`。
   - 若未传入 `targetValues`，直接返回 `actualValues`。
2. `roll()` 改为 `async`，并在 `demo-3d/index.tsx` 的 `handleDiceRoll` 中保持 `await`。
3. 静止检测期间仍需继续 `requestAnimationFrame` 物理 step；建议把 `animate()` 的主循环与 `roll()` 的等待状态机解耦，或在 `animate()` 内轮询 settlement。

### 3.3　3D 引擎应使用 convexHull 碰撞体

- **文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:119-152`
- **Why**：`ColliderDesc.ball(0.5)` 把骰子近似为球体，滚动行为不真实，且无法体现各面差异。设计文档 §3 步骤 10 明确要求用 `ColliderDesc.convexHull(Float32Array)` 从 `geometry.attributes.position.array` 构造。

**Before**
```ts
const collider = RAPIER.ColliderDesc.ball(0.5)
  .setRestitution(0.5)
  .setFriction(0.6)
world.createCollider(collider, body)
```

**After**
```ts
const positions = geo.attributes.position.array as Float32Array
const collider = RAPIER.ColliderDesc.convexHull(positions)
if (!collider) {
  throw new Error(`无法为 d${params.sides} 构造凸包碰撞体`)
}
collider.setRestitution(params.physics?.restitution ?? 0.5)
collider.setFriction(params.physics?.friction ?? 0.6)
world.createCollider(collider, body)
```

**注意事项**：
- `convexHull` 接收的是 `Float32Array`，直接传 `geometry.attributes.position.array`（已经是 Float32Array）即可。
- 若 convexHull 失败，按设计文档 §6 异常处理 3 应停止并上报，不可自行降级为 box/ball。
- 经验：Three.js 0.184 的 `BoxGeometry`/`OctahedronGeometry`/`DodecahedronGeometry`/`IcosahedronGeometry` 及手写 d10 几何体的顶点都是凸包，convexHull 应能成功。

### 3.4　Rapier 初始化未在 `Dice3DEngine` 内保证

- **文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:1-4`、`frontend/src/pages/demo-3d/index.tsx:22-26`
- **Why**：`Dice3DEngine.ts` 直接 `import RAPIER from '@dimforge/rapier3d-compat'` 并创建 `RAPIER.World`，但 `RAPIER.init()` 全局只能调用一次。`demo-3d/index.tsx` 有 `ensureRapierReady` 单例，但 `Dice3DEngine` 没有复用。当前代码「恰好」能在 rigid 场景先初始化 Rapier 后切到 dice 时工作，但如果用户直接以 dice 场景刷新页面或从外部路由进入，`createDice3DEngine` 会在未 init 的 Rapier 上操作，导致崩溃。

**建议方案**：把 `ensureRapierReady` 单例抽到共享模块（例如 `frontend/src/game/physics/rapierInit.ts`），`demo-3d/index.tsx` 与 `Dice3DEngine.ts` 都 import 该单例。`createDice3DEngine` 在函数开头 `await ensureRapierReady()`。

**After（共享文件骨架）**
```ts
// frontend/src/game/physics/rapierInit.ts
import RAPIER from '@dimforge/rapier3d-compat'

let initPromise: Promise<typeof RAPIER> | null = null
export function ensureRapierReady(): Promise<typeof RAPIER> {
  if (!initPromise) initPromise = RAPIER.init().then(() => RAPIER)
  return initPromise
}
```

### 3.5　demo-3d 初始进入「骰子演示」时不自动创建引擎

- **文件**：`frontend/src/pages/demo-3d/index.tsx:242-260`、`276-279`
- **Why**：`useEffect` 只在 `sceneType === 'rigid'` 时调用 `startEngine()`；`sceneType === 'dice'` 时什么也不做。用户首次选择 dice 场景会看到空白画布，必须点「复位」才能初始化。

**After**
```ts
useEffect(() => {
  let cancelled = false
  const init = async () => {
    await ensureRapierReady()
    if (cancelled) return
    if (sceneType === 'rigid') {
      startEngine()
    } else {
      startDiceEngine() // 新增：按面板状态创建 Dice3DEngine
    }
  }
  init()
  return () => {
    cancelled = true
    engineRef.current?.stop()
  }
}, [sceneType])
```

并新增 `startDiceEngine()`，从面板读取 `count`/`sides`/`theme`/`physics` 后调用 `createDice3DEngine`。

### 3.6　2D/3D 面板参数未写入 registry / 未传给引擎

- **文件**：`frontend/src/pages/demo-2d/index.tsx:244-285`、`frontend/src/pages/demo-3d/index.tsx:276`、`Dice2DPanel.tsx`、`Dice3DPanel.tsx`
- **Why**：面板状态与 Scene/Engine 割裂。`Dice2DPanel` 的 `count`、`Dice3DPanel` 的 `count` 都是局部 state，从未被消费。

**推荐重构方向**：
1. 把 `count`、`sides`、`theme`、`physics`、`animDuration` 提升到 `Demo2DPage` / `Demo3DPage` 的 state。
2. 通过 `game.registry.set('diceCount', count)` 等方式在创建游戏前写入 registry（2D），或在创建引擎时直接作为参数传入（3D）。

### 3.9　2D Sprite Scene 未在 `destroy()` 中清理事件监听

- **文件**：`frontend/src/pages/demo-2d/DiceSpriteScene.ts:25-31`、`DiceMatterScene.ts:41-44`
- **Why**：Scene 在 `create()` 中注册 `this.game.events.on('dice-roll', ...)`，但没有 `shutdown`/`destroy` 回调去 `off`。切换 demoType 时 `game.destroy(true)` 会销毁游戏，但 `game.events` 是全局对象；若以后把 Scene 复用到同一 Phaser.Game 实例（例如不销毁游戏只切 Scene），会造成事件堆积。

**建议**：在 `create()` 内保存 handler 引用，并在 `shutdown` 中注销：
```ts
private rollHandler?: (presetValues?: number[]) => void

create() {
  this.rollHandler = (presetValues?: number[]) => this.rollDice(presetValues)
  this.game.events.on('dice-roll', this.rollHandler)
}

shutdown() {
  if (this.rollHandler) this.game.events.off('dice-roll', this.rollHandler)
}
```
