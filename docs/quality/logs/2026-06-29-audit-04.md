## 3. 🚨 新发现缺陷详细说明

### 🔴 N-1（P1）：随机模式 slerp 校准到 DiceRoller 随机值，物理滚动沦为纯装饰

**文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:185-245`（`roll` + `finishRoll`）

**现象**：
- 随机模式（无 `presetValues`）下，`targetValues = roller.roll(...).values`（DiceRoller 的 crypto 随机值）
- 物理滚动静止后，`actualValues = meshes.map(mesh => getUpFace(...))`（物理朝上面）
- `needsCorrection = targetValues.some((tv, i) => tv !== actualValues[i])` — DiceRoller 随机值与物理朝上面几乎不可能一致，**恒为 true**
- 所有骰子被 `correctDiceOrientation` slerp 到 DiceRoller 的随机值
- 最终 `resolve(targetValues)` — 返回 DiceRoller 值，**丢弃物理结果**

**用户可见效果**：骰子物理落地显示某面（如 2），然后可见地旋转跳到另一面（如 3）。物理滚动完全无意义。

**设计偏差**：
- `docs/dice_implementation_plan.md` §3 步骤 10 ⑦："若 **presetValues** 且实际值不符，用 `correctDiceOrientation`" — 仅预设模式 slerp
- `docs/quality/logs/2026-06-29-audit-03.md` §3.2 After 骨架："若未传入 targetValues，**直接返回 actualValues**"

**Before（当前代码，有 bug）**：
```ts
// Dice3DEngine.ts:188-190
const targetValues = presetValues
  ? presetValues.slice()
  : roller.roll({ count: params.count, sides: params.sides }).values  // ← 随机模式也设 targetValues

// finishRoll:140-163
const actualValues = meshes.map((mesh) => getUpFace(mesh.quaternion, params.sides))
const needsCorrection = targetValues.some((tv, i) => tv !== actualValues[i])  // ← 随机模式恒 true
if (needsCorrection) {
  // slerp 到 targetValues（DiceRoller 随机值）...
  resolve(targetValues)  // ← 返回 DiceRoller 值，非物理值
}
```

**After（修复方案）**：
```ts
// roll() — 随机模式不调 DiceRoller，preset 模式才设 targetValues
const isPreset = !!presetValues
const targetValues = isPreset ? presetValues!.slice() : null

// finishRoll()
const actualValues = meshes.map((mesh) => getUpFace(mesh.quaternion, params.sides))

if (isPreset && targetValues) {
  // 仅预设模式：slerp 校准到 targetValues
  const needsCorrection = targetValues.some((tv, i) => tv !== actualValues[i])
  if (needsCorrection) {
    let correctionsRemaining = 0
    for (let i = 0; i < meshes.length; i++) {
      if (targetValues[i] !== actualValues[i]) {
        correctionsRemaining++
        const targetQ = getTargetQuaternion(params.sides, targetValues[i])
        correctDiceOrientation(meshes[i], targetQ, 300, () => {
          correctionsRemaining--
          if (correctionsRemaining === 0) resolve(targetValues)
        })
      }
    }
  } else {
    resolve(targetValues)
  }
} else {
  // 随机模式：直接返回物理结果
  resolve(actualValues)
}
```

**验收**：
1. 随机模式投掷 → 骰子物理落地后**不跳转**，结果 = 物理朝上面
2. 预设模式输入 "3,5" → 骰子物理落地后 slerp 校准到 3 和 5
3. 随机模式多次投掷，结果分布应接近均匀（物理随机性）

### 🔴 N-2（P1）：d10 面贴图映射错误——indexed 几何体与 matchFaceNormal 不兼容

**文件**：`frontend/src/game/dice/geometry.ts:181-234`（`createD10Geometry`）、`geometry.ts:313-322`（`applyFaceTextures` 公共分支）、`geometry.ts:326-348`（`matchFaceNormal`）

**根因**：
- d10 是唯一使用 `geo.setIndex(indices)` 的几何体（line 224，indexed）
- d8/d12/d20 通过 `ensureNonIndexed()` 转为 non-indexed 后才 `addFaceGroups`
- `applyFaceTextures` 公共分支调用 `matchFaceNormal(posAttr, g.start, faceDefMap)`
- `matchFaceNormal` 用 `posAttr.fromBufferAttribute(posAttr, startIndex)` 直接读顶点数据——**仅对 non-indexed 几何体正确**
- d10 的 `g.start` 是**索引范围起点**（0, 6, 12, 18, ...），不是顶点范围起点
- d10 只有 12 个顶点（index 0-11），`g.start=12` 及以后读越界 → `getX(12)` 返回 `undefined`

**验证结果**（Node 脚本 + jsdom 测试确认）：
```
d10 has index: true
d10 position count: 12  (仅 12 顶点)
d10 groups: [{start:0,count:6,mi:0}, {start:6,count:6,mi:1}, {start:12,count:6,mi:2}, ...]
group[2].start = 12 → vertex 12 不存在（越界）
```

**影响**：
- d10 groups 0-1 读到错误顶点（pole/pole/ring 而非面三角形），法向量错误
- d10 groups 2-9 读越界顶点，法向量为 NaN，`matchFaceNormal` 回退到默认值 1
- **d10 在浏览器中面值贴图完全错位**（大部分面显示 1）

**注意**：此 bug 是原始实现遗留（audit-03 未检出，因 audit-03 聚焦 d12）。D-1 删除 d12 特殊分支后 d10 仍走公共分支，bug 未被触发修复。

**修复方案**（二选一，推荐方案 A）：

**方案 A：d10 转 non-indexed（与其他多面体一致）**
```ts
// geometry.ts createD10Geometry 末尾，return 前转 non-indexed
function createD10Geometry(size: number): THREE.BufferGeometry {
  // ... 现有顶点+索引构造 ...
  const geo = new THREE.BufferGeometry()
  geo.setAttribute('position', new THREE.Float32BufferAttribute(positions, 3))
  geo.setIndex(indices)
  geo.computeVertexNormals()
  // 转 non-indexed：expand index → vertex，groups 自动从 index-range 转 vertex-range
  const nonIndexed = geo.toNonIndexed()
  // toNonIndexed 后 groups 已转为 vertex-based，但可能丢失，需重建
  nonIndexed.clearGroups()
  const vertsPerFace = 6  // 每面 2 三角形 × 3 顶点 = 6 顶点
  for (let i = 0; i < 10; i++) {
    nonIndexed.addGroup(i * vertsPerFace, vertsPerFace, i)
  }
  return nonIndexed
}
```

**方案 B：matchFaceNormal 支持 indexed 几何体**
```ts
function matchFaceNormal(
  geometry: THREE.BufferGeometry,
  startIndex: number,
  faceDefMap: Map<number, THREE.Vector3>,
): number {
  const posAttr = geometry.getAttribute('position') as THREE.BufferAttribute
  const index = geometry.index
  // indexed：通过 index buffer 取实际三角形顶点
  const i0 = index ? index.getX(startIndex) : startIndex
  const i1 = index ? index.getX(startIndex + 1) : startIndex + 1
  const i2 = index ? index.getX(startIndex + 2) : startIndex + 2
  const a = new THREE.Vector3().fromBufferAttribute(posAttr, i0)
  const b = new THREE.Vector3().fromBufferAttribute(posAttr, i1)
  const c = new THREE.Vector3().fromBufferAttribute(posAttr, i2)
  // ... 后续不变 ...
}
```

**验收**：
1. 新增测试：d10 `applyFaceTextures` 返回 10 个材质，每个对应不同 faceValue（1-10 全覆盖）
2. 新增测试：d10 geometry `geo.index` 为 null（方案 A）或 `matchFaceNormal` 正确处理 indexed（方案 B）
3. 浏览器实测：3D d10 投掷后，朝上面数字与 `getUpFace` 返回值一致

**问题**：`onSidesChange={setDiceSides}` 只更新 React state，未同步 `game.registry.set('diceSides', v)`。Scene 在 `create()` 时读 registry，如果用户改 sides 后不重置游戏，Scene 仍用旧值。

**实际影响**：2D 面板非 d6 选项全 disabled（`Dice2DPanel.tsx:9-12`），用户无法实际切换 sides，此路径不可达。

**修复**（防御性）：
```tsx
onSidesChange={(v) => {
  setDiceSides(v)
  gameRef.current?.registry.set('diceSides', v)
}}
```

### N-5（P3）：3D 投掷力 y 分量为负（向下冲击）

**文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:231-234`

**问题**：
```ts
body.applyImpulse(
  { x: (Math.random() - 0.5) * throwF * 0.2, y: -throwF * 0.13, z: (Math.random() - 0.5) * throwF * 0.2 },
  true,
)
```
`y: -throwF * 0.13` = -1.95（throwForce=15），向下冲击。骰子起始 y=6-8，向下冲击加速坠落，效果像"砸地"而非"抛掷"。

**影响**：UX 不自然，但不影响结果正确性。骰子滚动距离短，静止快。

**修复建议**：
```ts
// 改为向上抛起 + 水平随机方向
body.applyImpulse(
  { x: (Math.random() - 0.5) * throwF * 0.3, y: throwF * 0.1, z: (Math.random() - 0.5) * throwF * 0.3 },
  true,
)
```

### N-6（P3）：finishRoll 在 slerp 完成前置空 pendingRoll

**文件**：`frontend/src/pages/demo-3d/Dice3DEngine.ts:164`

**问题**：`pendingRoll = null`（line 164）在 slerp 校准 `onComplete` 回调之前执行。如果用户在 slerp 动画期间点"复位"，`stop()` 会 dispose mesh，但 `correctDiceOrientation` 的独立 RAF 仍在运行，对已 dispose 的 mesh 设置 quaternion（不报错但无意义）。

**实际影响**：UI 的"投掷"按钮在 rolling 期间 `loading={rolling}` 禁用，无法触发新 roll。"复位"按钮未禁用，但 slerp 对 disposed mesh 写 quaternion 不崩溃，仅浪费 RAF。属理论问题。

**修复**（可选）：
```ts
// clearDice 中取消所有进行中的 slerp
private slerpCancellers: (() => void)[] = []
// correctDiceOrientation 包装，返回 cancel 函数
// clearDice 调用所有 cancel
```
或更简单：`stop()` 中 `pendingRoll = null` 已阻止 `finishRoll` 再触发，slerp 的 `onComplete` 仍会调 `resolve`（Promise 已 await），不会卡死。可接受不修。
