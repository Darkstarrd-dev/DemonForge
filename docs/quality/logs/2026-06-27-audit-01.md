一个 `AppState`（`appStore.ts:138-271`）同时持有 M1–M5 业务数据、节点测试运行态、导入会话、角色交流配置、Provider 配置、UI 偏好，外加内联三套持久化引擎（business / settings / importSession）、引导加载 `bootstrapStore`、向后兼容迁移。`subscribe` 里手写 13 项 `s.xxx === prev.xxx` 脏检查（`:885-905`），新增业务字段需多处手工同步 → OCP 违反与变更放大器。

### 🔴 P0-2：SSE 流解析逻辑在前端 11 个文件中重复 34 次
`services/real/` 下 chat/extract/creation/image/gptImage/simulate/roleChat/xaiImage/llm 几乎每个文件各写一遍 `getReader()+TextDecoder+split('\n\n')+JSON.parse`；仅 `llm.ts` 内 `streamSingleChapter`(`:168-205`) 与 `streamBatch`(`:328-368`) 就重复两份。对比后端集中于唯一的 `sseHelper.ts`。每修一个解析 bug 要改 9 处。

项目无任何 `*.test.ts` / 测试框架（`frontend/package.json` scripts 仅 dev/build/lint/preview）。同时 `node-test/index.tsx.backup`（1114 行）被 git 跟踪，与现役 2355 行 `index.tsx` 并存。复杂调度器（熔断/重试/热更新）与精密持久化竞态逻辑靠手测+注释维持正确性，不可持续。

## 3. 🔍 逐项审计与重构建议

### 3.1　上帝 Store 拆分（zustand slices 模式）
- **文件**：`frontend/src/store/appStore.ts:138-271`、订阅脏检查 `:885-905`
- **Why**：状态定义 / action / 持久化 / 网络 / 引导 / 迁移六职责混于一文件；新增字段易漏加脏检查比较导致"改了不落库"。

**After（要点）**：按域切 `booksSlice / nodeTestSlice / roleChatSlice / uiPrefsSlice / providerSlice`，用 `StateCreator` 组合；持久化抽到 `store/persistence.ts`，以选择器声明式注册，告别手写 13 项比较。

### 3.2　SSE 解析抽公共 async generator
- **文件**：`frontend/src/services/real/llm.ts:168-205, 328-368` + 其余 8 个 `real/*.ts`
- **Why**：见 P0-2；且 `JSON.parse(data)` 无保护（`:185,345`），截断块会抛到外层 catch。

**After（要点）**：`services/sse.ts` 导出 `async function* parseSSE(body, signal)`，内部统一帧解析 + `safeJsonParse` 兜坏帧 + `reader.releaseLock()`；调用方收缩为 `for await (const {event,data} of parseSSE(...))`。约 200 行重复 → 1 处实现。

### 3.3　350 行巨型闭包调度器 → `CleanScheduler` 类
- **文件**：`frontend/src/services/real/llm.ts:392-744`（`startCleanQueue`）
- **Why**：10+ 闭包状态 Map/Set + 全部调度逻辑（dequeue/熔断/避让/重试/热更新/模型切换）内联，最该测的核心逻辑无法单测。

**After（要点）**：状态变字段、逻辑变方法，熔断独立成 `NodeCircuitBreaker`；对外 `startCleanQueue = (...args) => new CleanScheduler(...args).start()`，调用方零改动。

### 3.4　真实 Bug：每日额度重置用 UTC 日期，注释却写"本地自然日"
- **文件**：`frontend/src/store/appStore.ts:580`（`consumeProviderUsage`）；契约注释 `types.ts:369`
- **Why**：`new Date().toISOString().slice(0,10)` 取 UTC 日期。UTC+8 每日 00:00–08:00 额度不重置（UTC 仍前一日），用户清晨用不满当日额度。确定性错误。

**Before**：`const today = new Date().toISOString().slice(0, 10)`
**After**：抽 `utils/date.ts` 的 `localDateKey()`（用 `getFullYear/getMonth/getDate` 拼本地日期），调用处替换；并补跨日/跨时区单测。

### 3.5　胖组件群
- **文件**：`node-test/index.tsx`（2355 行 / 39 hooks）、`settings/index.tsx`（1868 行 / 24 hooks）
- **Why**：单组件 39 处 hooks 远超阈值；settings 从状态接口反推承担 8+ 类职责。
- **进展**：node-test 已拆出 HistoryList/DebugInfoPanel/SystemPromptEditor/SessionSidebar/ResultImage，`docs/` 有 5 份重构进度文档 → 作者已在重构，未完成。
- **建议路径**：抽 `hooks/useInferenceSession`、`hooks/useNodeTestForm`，拆 `panels/TextParamsPanel`、`panels/ImageParamsPanel`、`ChatTranscript`，`index.tsx` 仅留布局编排（<300 行）；建立缺失的 `frontend/src/hooks/` 层。

### 3.6　服务层 mock→real 迁移停在半成品
- **文件**：`frontend/src/services/api.ts:7-33`
- **Why**：mock/real 混杂导出，`checkConsistency`(mock) 与 `checkConsistencyReal`(real) 靠重命名并存，调用方易错；`generateChapterDraft` 仍 mock。
- **建议**：定义统一 interface 契约，清理 mock 残留，让 mock/real 成为实现细节而非两个可见名字。

### 3.7　其余记录项（中低优先级）

| 位置 | 问题 | 建议 |

| `main.tsx:67` | M1（1226 行）未懒加载，demo-3d/2d 却懒加载 | 懒加载策略按体积统一 |

> **被证伪的疑虑（澄清区）**
> - **疑似内存泄漏 → 无**：实测 `node-test/index.tsx` 的 `setInterval`(`:160-161`)、`addEventListener('paste')`(`:569-570`)、`AbortController.abort`(`:581-588`) 清理均成对正确。该组件问题是体积膨胀，非泄漏。
> - **疑似全量同步丢数据 → 否**：`db.ts:136-158` `syncAll` 有事务 + 预编译 + `ON CONFLICT upsert`；问题仅在扩展性（全量传输粒度），非正确性。属可延后优化项。
> - **疑似 SQL 注入 → 无**：表名来自白名单常量 `ENTITIES`，id 走参数绑定（`db.ts:143,174`），无注入面。

> **值得保持的亮点（重构勿破坏）**
> - `db.ts:136-158` 事务 + 预编译 + 白名单表名；WAL + busy_timeout（`:94-95`）。
> - `sseHelper.ts:20-21` 监听 `reply.raw` 的 `close` → `ac.abort()` 中止上游，连接不泄漏；陷阱注释（必须监听响应而非 req.raw）正确。
> - `ErrorBoundary.tsx` 规范的 class 错误边界 + resetKey 重挂载。
> - 注释质量极高（踩坑记录详尽）——但需固化为单测才能在回归时报警。
