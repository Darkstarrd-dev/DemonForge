## 2. 🚨 核心问题清单（High Priority）

### 🔴 P0-1：role-chat 本地模式在 Electron 打包版必然失效（后端读错配置路径）
`server/src/routes/chat.ts:45` 用 `new URL('../data/settings.json', import.meta.url).pathname` 直接读**源码旁**配置，是全后端**唯一**绕过统一数据目录抽象 `getAppDataDir()` 的地方（对比 `routes/settings.ts:7`、`routes/importSession.ts:6` 均走 `getAppDataDir()`，优先 `NOVELHELPER_DATA_DIR`）。开发环境两路径碰巧都指向 `server/src/data/` 故可用；**打包版数据目录为 `~/.novelhelper/`，chat.ts 仍读打包目录内的 data → `settings.providers` 为空 → `provider` 必然 `find` 不到 → 返回 404「节点不存在」**。即使用户在打包版手动配置了节点（写入 `~/.novelhelper/settings.json`），本地角色对话仍读不到。影响：role-chat 本地模式（核心功能之一）在生产形态完全不可用。

### 🔴 P0-2：role-chat 自动循环——流不可中断 + 卸载后继续运行 + 历史快照陈旧
`pages/role-chat/index.tsx` 的自动循环存在三个叠加问题：
- **流不可中断**：`respondLocal:118` 调 `sendLocalRoleMessage(...)` **未传 `signal`**，而 `services/real/roleChat.ts:76` 明明支持 `signal?` 形参。点「停止循环」只能在两次响应之间通过 `abortRef`（布尔）生效，**无法打断正在进行的 SSE 流**。
- **卸载无清理**：整个页面**没有 `useEffect` 卸载钩子**去置 `abortRef.current=true`。用户在循环中切走页面 → `runAgentLoop` 的 `while (!abortRef.current)`（`:325`）继续在后台跑 → 持续向上游 LLM 发请求（烧 token）+ 对**已卸载组件** `setMessages`/`setParticipants`（React 警告）。
- **历史快照陈旧（疑似正确性 bug）**：`runAgentLoop` 是长 async 循环，经 `respondLocal:121` 引用的 `messages` 是**启动循环那一刻的闭包快照**（非函数式读取）。循环期间各 Agent 的 LLM 请求只携带启动时的对话历史，**互相看不到对方在循环中的新发言**——UI 上消息正常累积（`setMessages` 用函数式更新），但模型上下文停滞，自动循环退化为"各 Agent 对初始消息各说各话"，削弱"多角色交流"的核心价值。机理依据 React 闭包语义可确认；用户感知程度建议实测验证。

## 3. 🔍 逐项审计与重构建议

### 3.1　chat.ts 数据目录读取（对应 P0-1）
- **文件**：`server/src/routes/chat.ts:45-58`
- **Why**：硬编码 `new URL('../data/settings.json')` 与全项目 `getAppDataDir()` 单一真相源不一致；生产数据目录 `~/.novelhelper/` 下读不到 providers。

**Before**
```ts
const settingsPath = new URL('../data/settings.json', import.meta.url).pathname
const settingsText = readFileSync(settingsPath, 'utf-8')
const settings = JSON.parse(settingsText)
provider = settings.providers?.find((p: any) => p.id === nodeId)
```
**After**
```ts
import { readSettings } from './settings' // 已有：走 getAppDataDir() + .bak 回退
const providers = (readSettings().providers ?? []) as ProviderNode[]
const provider = providers.find((p) => p.id === nodeId)
```
> 顺带消除 `card: any`(`:31`)/`provider: any`(`:46`) 两处后端裸 `any`（B-7）。

### 3.2　role-chat 自动循环生命周期（对应 P0-2）
- **文件**：`pages/role-chat/index.tsx:55, 103-168, 313-369, 372-413`
- **Why**：见 P0-2。`abortRef` 用布尔而非 `AbortController`，流不可中断；无卸载清理；闭包捕获陈旧 `messages`。

**After（要点）**：
1. 引入真 `AbortController`（替代布尔 `abortRef`），`respondLocal` 透传 `signal` 给 `sendLocalRoleMessage`。
2. 加 `useEffect(() => () => { acRef.current?.abort(); abortRef.current = true }, [])` 卸载清理。
3. 用 `messagesRef`（`useRef` 镜像 + 每次 `setMessages` 同步）或在 `respondLocal` 内函数式读取最新 messages，喂给后端的 `conversationHistory` 取 `messagesRef.current`。

### 3.3　持久化脏检查未声明式化（承接 audit-01 P0-1，A-7 残留）
- **文件**：`store/persistence.ts:277-299`（业务 12 项 `===`）、`:302-324`（设置 20 项 `===`）；与 `businessPayload:19-32`、`settingsPayload:158-179` 三处需手工同步
- **Why**：audit-01 P0-1 的核心担忧是"手写脏检查 = OCP 违反 + 变更放大器，新增字段易漏比较导致改了不落库"。A-7 把上帝 Store 拆成 6 切片（主目标达成），但脏检查只是从 `appStore.ts` **整体搬到** `persistence.ts`，**未改成声明式**——设置项还从 13 增到 20，脆弱面更大。

**After（要点）**：以 payload 的键集为单一真相，自动生成比较，加字段只改 payload 一处：
```ts
const SETTINGS_KEYS = Object.keys(settingsPayload(useAppStore.getState())) as (keyof AppState)[]
useAppStore.subscribe((s, prev) => {
  if (!storeReady) return
  if (SETTINGS_KEYS.every((k) => s[k] === prev[k])) return
  // ...debounce POST
})
```

### 3.4　settings 组件内手写 SSE 解析（audit-01 P0-2 漏查的残留点）

**After**：`for await (const { event, data } of parseSSE(res.body)) { if (event==='delta'||event==='done') acc += (data as any).delta ?? (data as any).text ?? '' }`
