# M1 入库失败问题诊断指南

## 问题现象
入库时报错：`入库失败：Failed to fetch。数据未保存，请重试或检查章节内容大小`

## 已实施的修复措施

### 1. 增加超时控制（60 秒）
- **位置**: `frontend/src/store/appStore.ts` - `pushStore` 函数
- **修改**: 从 30 秒增加到 60 秒超时
- **原因**: 大数据量写入需要更长时间

### 2. 添加数据大小检测
- **JSON 序列化前检查**: 预先计算数据大小，记录到控制台
- **入库前警告**: 超过 30MB 会弹窗确认
- **控制台日志**: 显示 `[pushStore] 数据大小: X.XX MB`

### 3. 改进错误信息
- **超时**: 明确提示"请求超时（超过 60 秒）"
- **网络错误**: 显示具体的网络错误信息
- **后端错误**: 解析并显示后端返回的错误详情

## 诊断步骤

### 步骤 1：打开浏览器开发者工具
1. 按 F12 打开开发者工具
2. 切换到 **Console（控制台）** 标签
3. 尝试入库操作

### 步骤 2：查看控制台日志
在入库时应该看到：
```
[M1入库] 章节数: X, 总内容大小: X.XX MB
[pushStore] 数据大小: XX.XX MB
```

**如果数据大小超过 40MB**：
- 这可能是导致失败的主要原因
- 建议分批入库或清理旧数据

### 步骤 3：查看 Network（网络）标签
1. 切换到 **Network** 标签
2. 尝试入库
3. 查找 `/api/store` 请求
4. 查看状态码和响应

**可能的状态**：
- **413 Payload Too Large**: 数据超过后端 50MB 限制
- **500 Internal Server Error**: 后端处理错误
- **Failed (net::ERR_*)**: 网络层错误
- **Canceled**: 请求被中断（可能是超时）

### 步骤 4：检查数据库大小
```bash
ls -lh Z:/Playground/novelhelper/server/src/data/assets/novelhelper.db
```

**如果数据库超过 100MB**：
- 考虑清理旧数据或重建数据库
- 数据库过大会导致读写变慢

## 临时解决方案

### 方案 1：分批入库
不要一次性入库所有章节，而是分批进行：
1. 在 M1-Step2 分割章节时，只保留前 50 章
2. 入库后再导入下一批

### 方案 2：清空数据库重新开始
**警告：会删除所有已入库数据！**

1. 停止前后端服务
2. 删除数据库文件：
   ```bash
   rm Z:/Playground/novelhelper/server/src/data/assets/novelhelper.db
   ```
3. 重启服务
4. 系统会自动创建新的空数据库

### 方案 3：使用恢复出厂功能
在系统设置 → 备份与恢复 → 点击"恢复出厂"按钮
- 清空所有设置
- **保留业务数据**（书籍、章节不会删除）

### 方案 4：增加后端 bodyLimit
如果确认数据量确实很大且合理，可以增加后端限制：

编辑 `server/src/index.ts`:
```typescript
const app = Fastify({ 
  logger: true, 
  bodyLimit: 100 * 1024 * 1024  // 改为 100MB
})
```

## 常见原因分析

### 原因 1：数据库已有大量数据
- **现象**: 数据库文件 16MB+，入库失败
- **原因**: 每次入库需要同步整个数据库到后端
- **解决**: 清理旧数据或分批入库

### 原因 2：单次入库章节过多
- **现象**: 一次入库 100+ 章节
- **原因**: JSON 序列化后数据量过大
- **解决**: 分批入库，每次不超过 50 章

### 原因 3：章节内容过大
- **现象**: 单个章节超过 100KB
- **原因**: 原始文本未清理干净，包含大量垃圾字符
- **解决**: 在 M1-Step3 进行文本清理

### 原因 4：网络不稳定
- **现象**: 间歇性失败，有时成功有时失败
- **原因**: 本地网络或服务问题
- **解决**: 检查防火墙、杀毒软件，重启前后端服务

### 原因 5：后端未启动或端口被占用
- **现象**: 始终报 `Failed to fetch`
- **原因**: 后端服务未运行
- **解决**: 确认后端在 `http://localhost:3000` 正常运行

## 后端日志检查

查看后端控制台输出，寻找错误信息：
```
ERROR: ...
```

常见后端错误：
- **SQLITE_BUSY**: 数据库被锁定（多进程访问）
- **SQLITE_CORRUPT**: 数据库文件损坏
- **Out of memory**: 内存不足

## 测试建议

### 测试 1：入库极小数据
1. 在 M1-Step2 只保留 1-2 章
2. 尝试入库
3. 如果成功 → 确认是数据量问题

### 测试 2：检查后端连通性
在浏览器访问：
```
http://localhost:3000/api/store
```
应该返回 JSON 格式的业务数据

### 测试 3：使用 curl 测试
```bash
curl -X POST http://localhost:3000/api/store \
  -H "Content-Type: application/json" \
  -d '{"books":[],"chapters":[]}'
```
应该返回 `{"ok":true}`
